   - 这个顺序很重要！如果你先扩展再改写，可能会导致语义漂移

---

### 【更完美的回答范例】

> "我的实现中，查询改写和查询扩展是'主路+旁路'的关系：
> - **主路（默认）**：用户query先经过 `process()` 做改写（修正错别字、补充上下文），然后直接进向量检索
> - **旁路（复杂场景）**：当用户问题比较模糊时，调用 `process_with_expansions()`，它会先改写再扩展出3-5个同义词query，对每个query分别检索后再去重合并

> - 这个设计的好处是：简单问题低开销，复杂问题高召回，兼顾了效果和性能"

## 第三问：如果继续把系统补成更接近生产可用，你会怎么规划？

### 面试官提问：

如果现在要继续补下面 5 项能力：
1. 元数据过滤：增加 `title`、`section_path`、`doc_type`、`sheet_name`、时间戳等结构化字段
2. 查询改写/拓展不再只依赖本轮 query，要支持 history-aware rewrite
3. 长期记忆，或者较早轮次的 summary
4. 工具的重试、降级、熔断机制完善
5. 多租户隔离

你会怎么判断“能尽快完成”的边界？怎么拆解需求、给出设计方案，并制定验收标准和测试方案？

---

### 【高分回答范例】

我会先把这 5 项收敛成一个“一期可快速交付版本”，目标不是一次做成最终生产形态，而是先把主链路补完整，让检索、会话记忆、工具容错、租户隔离这四层都能在现有架构上闭环。

#### 1. 先看现状，再定边界

我会先明确当前代码里的几个真实缺口：

- 当前 chunk metadata 主要只有 `source / chunk_index / total_chunks / strategy`，还不能按 `title`、`doc_type`、`sheet_name`、时间范围做过滤。
- 当前 query rewrite 主要挂在 `RAGService` 上，但主对话链路更常走的是 `ChatAgent -> query_internal_docs -> VectorStore.search()`，这条链路实际上还没有统一吃到 rewrite / expansion / history。
- 当前 `SessionStore` 只保留最近 6 轮对话，旧轮次是直接裁剪，没有 summary，也没有长期记忆。
- 当前工具调用是 Agent 里直接 `tool.ainvoke()`，缺少统一的 retry、fallback、circuit breaker；一旦外部工具波动，容易把整条链路打断。
- 当前系统本质上还是单租户：Milvus collection、Redis session key、上传目录、API 请求都没有 tenant 维度。

所以我不会把这 5 项拆成 5 个孤立功能，而是会抽象成 4 个改造面：

1. 检索上下文层：统一 query、history、summary、filters、tenant
2. 会话记忆层：最近消息 + 滚动 summary
3. 工具治理层：统一重试、降级、熔断
4. 租户隔离层：检索、会话、上传、删除全部带 tenant

---

#### 2. “尽快完成”的边界怎么定

我会把一期边界控制在 5 到 7 个工作日，只做“逻辑隔离 + 共享检索上下文 + 基础容错”，先不做过重的生产化能力。

**一期纳入范围：**

- 元数据字段补齐，并支持白名单过滤
- history-aware rewrite，并且让 Agent 主链路也能用上
- 会话滚动 summary，保留最近 N 轮 + 早期摘要
- 外部工具统一 retry / fallback / circuit breaker
- 多租户逻辑隔离：同 collection、同 Redis、不同 tenant 过滤和 key 前缀

**一期不纳入范围：**

- 物理级多租户隔离，比如“一租户一 collection / 一 Redis DB”
- 真正的长期记忆向量化检索
- 复杂权限模型、RBAC、租户级配额
- 分布式熔断状态同步
- 自动从文档正文中抽取业务时间字段并做高精度时间语义解析

这样做的原因是：一期先解决“能用、可测、不串租户、不因上下文缺失而失真、不因工具抖动而整体失败”，先把系统从 Demo 拉到可交付。

#### 3. 设计方案

##### 3.1 元数据过滤

我会把 metadata 扩成一个统一结构，首批建议如下：

```json
{
  "tenant_id": "acme",
  "doc_id": "acme:cpu_high_usage.md",
  "source": "cpu_high_usage.md",
  "title": "CPU 使用率过高排查",
  "doc_type": "markdown",
  "sheet_name": null,
  "section_path": "概述/排查步骤/常见原因",
  "ingested_at": 1711900800,
  "chunk_index": 3,
  "total_chunks": 12,
  "strategy": "markdown"
}
```

设计上我会注意两点：

- `section_path` 一期先存成字符串，不做数组结构，避免过滤表达式复杂化。
- 时间字段一期先用 `ingested_at`，即入库时间戳；后续如果业务需要，再补 `content_timestamp`。

过滤能力一期只支持白名单：

- 精确匹配：`tenant_id / doc_type / sheet_name / source`
- 包含匹配：`title / section_path`
- 时间范围：`ingested_at_from / ingested_at_to`

不做任意 DSL，让后端把 filter object 转成安全的 Milvus expr，避免注入和表达式失控。

##### 3.2 history-aware rewrite

我会新增一个统一的 `RetrievalContext`，里面包含：

- 当前问题 `query`
- 最近几轮 `recent_history`
- 历史摘要 `summary`
- 元数据过滤 `filters`
- 租户 `tenant_id`

然后把 query 改写改成两段：

1. Contextualize：把当前问题改写成“自包含问题”
2. Expand：再基于自包含问题做 expansion

例如用户问“这个怎么处理？”、“CPU 高是不是和刚才那个实例有关？”这种口语化问题时，先把指代补全，再做拓展；否则 expansion 很容易扩错方向。

关键点不是只改 `RAGService`，而是要把检索入口统一掉：

- `RAGService.generate_answer()` 走统一检索管线
- `query_internal_docs` 工具也走统一检索管线

这样主链路和回退链路才是一致的。

##### 3.3 长期记忆 / 较早轮次 summary

一期我不会直接做“长期记忆向量库”，因为太重。我会先做 rolling summary：

- Redis 里保留最近 `N` 轮原始消息
- 超出的早期消息不直接丢，而是合并进 `session_summary`
- 下一轮 query rewrite 和 answer generation 时，把 `summary + recent_history` 一起带入

这样成本可控，而且能明显改善“超过 6 轮就遗忘”的问题。

总结策略我会控制得很保守：

- 只总结事实、约束、已确认结论、未解决事项
- 不总结语气词，不总结无关寒暄
- 每次增量总结，而不是反复重写全部 summary

##### 3.4 工具的重试 / 降级 / 熔断

我会把工具治理做成统一包装层，而不是分散写在每个 tool 里。

一期策略：

- `retry`：仅对超时、网络抖动、5xx 做 2 到 3 次指数退避重试
- `fallback`：工具最终失败时返回标准化降级结果，不直接抛异常打断 Agent
- `circuit breaker`：同一工具连续失败达到阈值后，短时间内直接熔断，返回“工具暂不可用，请改走知识库/人工兜底”

这里要区分两类工具：

- 外部依赖型工具，如 Tavily、未来真实 Prometheus 查询：需要 retry + breaker
- 本地静态工具，如时间工具、mock log 工具：只需要异常兜底，不需要 breaker

##### 3.5 多租户隔离

一期我会做逻辑隔离，不做物理隔离。

具体做法：

- API 请求增加 `tenant_id`
- 上传目录改成 `uploads/{tenant_id}/...`
- Redis session key 改成 `tenant:{tenant_id}:session:{session_id}:...`
- 文档 metadata 强制写入 `tenant_id`
- 所有检索、删除、会话查询都默认附带 tenant filter

这样可以在不大改部署模型的情况下，先把“串租户”风险降下来。

#### 4. 需求拆解与落地顺序

我会按下面顺序做，确保每一步都可验证：

1. **接口与数据结构改造**
   - `ChatRequest`、上传接口补 `tenant_id`
   - 增加 `retrieval_filters`
   - metadata 结构补齐

2. **索引链路改造**
   - 上传时注入 `tenant_id / title / doc_type / sheet_name / ingested_at`
   - Markdown chunk 注入 `section_path`
   - VectorStore 支持 filter expr

3. **统一检索管线**
   - 抽 `RetrievalContext`
   - `RAGService` 与 `internal_docs_tool` 共享同一套 rewrite + expansion + search 逻辑

4. **会话记忆改造**
   - `SessionStore` 增加 summary key
   - 超长会话触发滚动总结

5. **工具治理改造**
   - 增加 `ToolExecutor` 或 `ResilientToolInvoker`
   - 统一 retry / fallback / circuit breaker

6. **回归与评测**
   - 补单测、集成测试、串租户测试、故障注入测试

#### 5. 验收标准

我会把验收标准写成“可观察、可回归”的形式，而不是只写抽象目标。

**5.1 元数据过滤**

- 上传 Markdown、Excel、CSV 后，chunk metadata 中可以看到 `title / doc_type / section_path / sheet_name / ingested_at / tenant_id`
- 检索接口或检索工具传入 filter 后，只返回符合过滤条件的文档
- 时间范围过滤生效，边界值可测

**5.2 history-aware rewrite**

- 对于带指代词、缩写、强上下文依赖的问题，系统会先改写成自包含问题
- Agent 主链路和 RAG 回退链路都使用同一套改写逻辑
- 对既有检索评测集，核心指标不能明显下降；对新增多轮数据集，`source_hit` 和 `hit@3` 要优于只看本轮 query 的 baseline

**5.3 summary / 长期记忆**

- 会话超过最近消息窗口后，旧消息不会直接丢失，而是沉淀到 summary
- 用户在第 8 到第 12 轮引用更早上下文时，系统仍能回答出核心约束和对象
- summary 更新后不会把历史事实篡改或互相矛盾

**5.4 工具容错**

- 外部工具出现一次性超时时会自动重试并恢复
- 连续失败达到阈值后，熔断生效，Agent 不再长时间等待该工具
- 工具失败时，系统返回降级结果或改走知识库，不出现整条链路 500

**5.5 多租户隔离**

- tenant A 上传的文档，tenant B 无法检索到
- tenant A 的 session 列表、summary、clear 操作不会影响 tenant B
- 删除文档、查看会话、检索结果都带 tenant 边界

#### 6. 测试方案

我会把测试分成 4 层：

**第一层：单元测试**

- metadata 组装测试：不同文件类型是否生成正确字段
- filter expr 生成测试：白名单字段、边界时间、空值处理
- query rewrite 测试：有 history 和无 history 两种模式
- summary 合并测试：旧消息压缩后关键信息不丢
- circuit breaker 状态机测试：closed -> open -> half-open

**第二层：集成测试**

- 上传 Markdown / Excel / CSV 后，按 `tenant_id + doc_type + sheet_name` 检索
- 多轮对话中先提实体，后面只用“它 / 这个 / 刚才那个实例”追问，验证 rewrite 后能正确召回
- 超过历史窗口后继续追问，验证 summary 生效
- 模拟 Tavily / Prometheus 超时和 5xx，验证 retry 和 fallback

**第三层：隔离与安全测试**

- 双租户并发上传、并发检索，验证 0 串租户
- 构造非法 filter 字段，验证后端拒绝并不会拼出危险表达式
- 清理 session / 删除文档时验证 tenant 边界

**第四层：回归评测**

- 复用现有 retrieval / generation eval，不允许核心指标明显回退
- 新增一组 conversation-aware retrieval cases，专门测口语化追问、缩写、跨轮指代
- 新增 fault-injection case，验证工具抖动时链路仍可用
